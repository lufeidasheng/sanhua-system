# 仓库资产三层边界

## 基本判定原则

目录归属只是一级规则。最终是否纳入主链测评、总控判断或主施工工单，必须以是否参与当前主链运行、测评或派工为准。

## Mainline（主链层）

定义：当前三花主链运行、调度、入口、模块、配置和治理取证直接依赖的资产。

默认目录：

- `core/`
- `entry/`
- `modules/`
- `config/` 中被当前启动链或调度链真实读取的配置
- `docs/` 中控制面与治理文档
- 少量当前主链治理测试

默认纳入：

- 主链测评：是
- 总控判断：是
- 主施工工单：可纳入，但仍以当前工单点名范围为准

例外规则：

- `tools/` 默认不属于 Mainline；只有被当前工单点名，或被主链验证实际使用的单个工具脚本，才按例外纳入。
- `docs/` 只纳入控制面、治理规则、债务记录、协调流等文档；压缩包、缓存和无关导出不纳入。
- `config/` 只纳入被当前启动链或调度链真实读取的配置；缓存和生成物不纳入。
- `tests/` 只纳入当前主链治理测试或当前工单明确点名的测试；不把全部测试目录自动视为 live 主链。

## Vendor / External（依赖层）

定义：外部依赖、模型资产、第三方示例、外部运行时数据或供应商资产。它们可以支撑主链运行，但不默认作为三花业务主链代码评判对象。

默认目录：

- `llama.cpp/`
- `juyuan_models/`
- `piper-master/`
- `third_party/`
- `deps/`
- `dependencies/`
- `external/`
- `ollama_*`
- `models/`

默认纳入：

- 主链测评：否
- 总控判断：仅在有 live 主链显式引用证据时纳入相关判断
- 主施工工单：默认不纳入，除非工单明确切到依赖或运行环境治理

例外规则：

- 如果 Mainline 中存在显式 import、配置引用、启动命令依赖或当前工单点名，则只把相关引用点纳入判断，不把整个 Vendor / External 目录纳入主链 blocker。
- 依赖层问题默认归为依赖、环境或模型资产风险；不得直接计入三花主链代码 blocker，除非存在可复现的 live 主链引用证据。

## Runtime / Legacy（运行/冻结层）

定义：运行产物、历史冻结资产、审计输出、回滚快照、虚拟环境、日志、录音、运行态数据和历史备份。它们可用于取证，但不默认作为当前主链施工对象。

默认目录：

- `_legacy_disabled/`
- `legacy/`
- `_audit*`
- `rollback_snapshots/`
- `logs/`
- `runtime/`
- `recordings/`
- `venv/`
- `.venv/`
- 运行态数据与历史备份

默认纳入：

- 主链测评：否
- 总控判断：仅作为运行取证或历史背景，默认不计入主链 blocker
- 主施工工单：默认不纳入，除非工单明确切到运行态取证、历史资产治理或环境治理

例外规则：

- `data/memory` 等运行态数据可作为运行态取证来源，但不作为代码测评对象。
- `.bak`、回滚快照、审计输出、日志、缓存和运行产物默认不作为主链质量风险计算对象。
- 如果 Runtime / Legacy 资产被当前 live 主链显式 import、执行、读取或被当前工单点名，则只评估该引用关系，不扩展为目录级治理。

## 主施工边界

- 不因资产分层规则自动移动目录。
- 不因资产分层规则自动删除历史资产。
- 不因资产分层规则自动修改 import、启动链或测评脚本。
- 每张施工工单仍以总控当轮允许触达文件为准。
