# 三花聚顶系统 · 总控协作流（V2）

## 核心模式
采用“1 个 GPT 总控 + 4 个 Codex 工位”协作推进：

- 总控：只做判断
- 1号：主施工位，只做执行
- 2号：记录/回填位，只做执行
- 3号：协议/骨架/审计位，只做执行
- 4号：验证/环境/证据位，只做执行

GPT 总控负责：
- 方向判断
- 工单切换
- 边界控制
- 验收判断

Codex 工位不负责：
- 最终拍板
- 越权切单
- 用隐形上下文代替文档控制面

docs 控制面负责显式状态记忆：
- `docs/NEXT_ACTION.md` 记录当前唯一主工单与切单状态
- `docs/TECH_DEBT.md` 记录需追踪的问题、状态与验证结果
- `docs/COORDINATION_FLOW.md` 记录协作规则与流转边界
- 协作默认不依赖隐形记忆共享，必须以 docs 为准

---

## 标准流转

### 阶段 1：定义工单
输入：
- 当前目标
- 所属主链
- 已知问题
- 风险边界

输出：
- 工单标题
- 主执行位
- 涉及文件
- 完成定义
- 最小验证方式

---

### 阶段 2：控制面回填
由 2号执行：
- 更新 `docs/NEXT_ACTION.md`
- 必要时更新 `docs/TECH_DEBT.md`
- 补充当前工单状态、目标、依赖关系
- 保持 docs 控制面与真实施工状态一致

---

### 阶段 3：施工与并行支持
由 1号执行：
- 做最小必要代码改动
- 保持主链闭环
- 留出明确验证点

由 3号并行：
- 做协议审视 / 骨架建议 / 审计结论

由 4号并行：
- 做环境验证 / 运行验证 / 日志证据收集

验证分层：
- L0：静态 / 编译级最小验证
- L1：定向纯测试
- L2：极小运行态取证

默认验证口径：
- 默认解释器：`./venv/bin/python`
- 默认最低验证：`PYTHONPYCACHEPREFIX=/tmp/pycache ./venv/bin/python -m py_compile <触达文件>`
- `pytest` 仅作为 L1 定向验证候选，不作为强默认
- 不把系统 `python` / `.venv` / GUI 全量启动 / 模型服务 / 音频设备 / `tools` 脚本写成默认验证前提

04号标准回包格式：
1. 验证等级
2. 实际执行的命令
3. 关键结果
4. 当前判断
5. 问题归因
6. 给总控的话

默认要求：
- 01/02/03/04 只执行各自工位职责
- 任何线程状态以 docs 控制面为准
- 不用“上一轮记忆里知道”替代当前文档事实

---

### 阶段 4：汇总与验收
汇总内容：
- 改了什么
- 验证了什么
- 还有什么没解决
- 是否达到 DoD
- 是否可进入下一工单

由 GPT 总控判断：
- 通过验收
- 继续补做
- 记技术债
- 切下一单

---

## 默认推进策略

### 第一优先级
主链闭环：
- 对话
- 记忆
- 动作路由
- 系统检测
- 模型切换
- 模块接管
- GUI 稳定

### 第二优先级
配套控制面：
- 文档归口
- 技术债可视化
- 验证证据化
- 施工边界化

### 第三优先级
探索项：
- 新协议
- 新骨架
- 新能力预研

---

## 工单切换条件

满足以下之一，可切下一单：

1. 当前工单已完成并通过最小验证
2. 当前工单无法继续推进，但已明确卡点并记入 TECH_DEBT
3. 当前线程上下文过长，需要先压缩总结后重开

---

## 线程过长处理
当上下文过长、信息密度下降、协作位混乱时：

1. 2号先做收尾摘要
2. 回填当前工单结论
3. 标注未完成项
4. 新线程只继承必要状态，不硬扛旧上下文
