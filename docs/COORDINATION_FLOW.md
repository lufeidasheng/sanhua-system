# 三花聚顶 · 总控协同推进流 v1

## 1. 文档目的

本文件用于固化三花聚顶系统当前的总控协作推进方式，保证多 Codex / 多线程推进时：

- 主线程不漂移
- 文件 owner 清晰
- 工单边界清晰
- 施工、验证、回填分工稳定
- 方向判断、验收判断有统一口径

本文件服务于“三花聚顶本地稳定版 v1 / 贾维斯本地内核雏形”。

---

## 2. 当前系统方向

三花聚顶不是普通聊天助手，而是一个：

- 本地优先
- 全天候常驻
- 具备长期记忆
- 具备聚感
- 具备行动控制
- 具备自我演化能力
- 未来支持多设备 / 家居接入

当前 v1 目标不是完整版贾维斯，而是：

- 贾维斯的本地内核雏形

当前 v1 优先做到：

- 知道用户是谁
- 知道自己当前状态
- 能稳定控制电脑核心能力
- 能记住历史
- 能持续接管动作
- 能稳定表达自己的变化

---

## 3. 当前组织结构

当前采用 4 个 Codex + 1 个总控中枢推进。

### 3.1 角色分工

#### 1号：主线施工位
负责当前唯一主施工工单。

职责：
- 修改主线代码
- 做最小验证
- 收敛结论
- 输出最小 diff 摘要
- 明确是否完成当前工单

默认特征：
- 是当前唯一主执行位
- 可以改 `.py`
- 默认占用主施工文件 owner

#### 2号：记录 / 回填 / 控制面位
负责控制面文档和结论回填。

职责：
- 维护 `docs/NEXT_ACTION.md`
- 维护 `docs/TECH_DEBT.md`
- 维护方向文档
- 回填已确认结论
- 切换当前主线程文档口径

默认特征：
- 默认不改 `.py`
- 不抢施工位
- 不承担主代码实施

#### 3号：协议 / 骨架 / 次主线审计位
负责协议审计、骨架设计、前置审计、平行子任务。

职责：
- 审计当前主链相关协议
- 设计骨架文件 / 接口草案
- 做不抢主文件的平行子任务
- 为 1号 提供前置结构判断

默认特征：
- 可做小范围代码改动
- 默认不抢 1号 主施工文件
- 更偏设计 / 协议 / 审计，而非主线大改

#### 4号：验证 / 环境 / 证据位
负责验证、环境、证据收集和非破坏性检查。

职责：
- 跑最小验证命令
- 定位环境问题
- 收集运行时证据
- 区分代码问题和环境问题
- 产出可用于验收的证据链

默认特征：
- 更偏验证和证据，不承担主线大改
- 默认不抢 1号 主施工文件

#### ChatGPT：总控中枢
负责方向判断、工单切换、边界控制、验收判断。

职责：
- 判断当前主线
- 指定下一张工单
- 控制文件边界和 owner
- 判断是否继续 / 收口 / 停止当前工单
- 区分主线问题、环境问题、协议问题、控制面问题
- 维护整体推进秩序

---

## 4. 核心协作规则

### 4.1 每张工单默认只有一个主执行位
- 一张工单只能有一个主施工 owner
- 默认由 1号 承担
- 其他位只能做平行支持、审计、验证、回填

### 4.2 同一文件同一时段只归一个施工位
- 同一主链文件不能被多个施工位同时改
- 若文件已被 1号 占用，3号 / 4号 默认不抢
- 若需切 owner，必须先由总控明确切换

### 4.3 2号默认不改 `.py`
2号固定为控制面位，只改 docs，不承担主线代码施工。

### 4.4 默认先主线，后支线
- 当前唯一主施工工单优先
- NAS、家居、视觉、多模型等方向，除非明确进入主线，否则只保留方向，不抢施工

### 4.5 默认先收边界，再扩能力
- 先收主桥
- 先收状态真相源
- 先收统一接口
- 先收运行环境边界
- 再扩新功能

### 4.6 默认不让 GUI 成为逻辑中心
- GUI 作为显示 / 输入 / 转发壳
- 主链逻辑优先收敛到统一总控和标准接口
- 默认不在 GUI 内继续扩写核心编排逻辑

---

## 5. 当前推进原则

始终遵循：

- 先打穿闭环，再扩展能力
- 模型 ≠ 系统
- 系统 = 调度 + 数据 + 记忆 + 执行
- 永远保持可控、可审计、可降级
- 先统一底座，再扩展功能
- 优先主链闭环，不做散功能推进
- legacy / 历史资产默认冻结，除非明确开工单
- 不把环境问题误判成代码主链问题
- 不让新方向抢当前底座统一主线

---

## 6. 标准工单推进流

当前默认流程如下：

方向判断  
→ 指定当前唯一主施工工单  
→ 1号施工  
→ 3号做平行审计 / 骨架支持  
→ 4号做验证 / 环境 / 证据  
→ 2号回填 `NEXT_ACTION` / `TECH_DEBT`  
→ ChatGPT 做验收判断  
→ 再切下一张工单

### 6.1 工单生命周期
每张工单默认经历：

1. 明确当前主线
2. 明确已确认事实
3. 明确工单名称
4. 明确目标
5. 明确允许触达文件
6. 明确禁止扩散边界
7. 实施 / 审计 / 验证
8. 输出结论
9. 回填控制面
10. 验收或切下一张工单

---

## 7. 输出规范

### 7.1 1号输出规范
1号完成后默认只输出：

1. 修改了哪些文件
2. 最小 diff 摘要
3. 验证命令
4. 验证结果
5. 是否完成当前工单
6. 若未完成，最小剩余故障点

### 7.2 2号输出规范
2号完成后默认只输出：

1. 修改了哪些文档
2. 回填了哪些内容
3. `NEXT_ACTION` 现在指向什么

### 7.3 3号输出规范
3号完成后默认只输出：

1. 审计了哪些文件 / 协议
2. 当前发现的结构判断
3. 对主线程的支持结论
4. 是否建议进入实施或保持预研

### 7.4 4号输出规范
4号完成后默认只输出：

1. 验证命令
2. 验证结果
3. 关键证据
4. 问题位于哪一层
5. 是否属于环境 / 权限 / 运行边界问题

---

## 8. 工单分类

为避免混乱，工单默认分为三类：

### A 类：施工工单
特点：
- 改 `.py`
- 做最小验证
- 收敛代码结论

默认 owner：
- 1号

### B 类：审计 / 验证工单
特点：
- 查协议
- 查证据
- 查环境
- 非破坏性验证
- 最小化或不改代码

默认 owner：
- 3号 / 4号

### C 类：控制面工单
特点：
- 回填台账
- 维护 `NEXT_ACTION`
- 维护 `TECH_DEBT`
- 同步方向文档

默认 owner：
- 2号

---

## 9. 当前默认禁止事项

默认不做以下事项，除非明确进入主线：

- NAS / storage_hub 实施
- 家居接入实施
- GUI 大重构
- 分布式系统
- 大范围 legacy 清理
- 训练基础大模型
- 不可控自动执行
- 多模型花活优先
- 任何破坏当前主链收口的散功能扩张

---

## 10. 当前默认优先级

当前优先级默认遵循：

1. 运行环境边界收口
2. 状态真相源
3. `SystemContext` 标准接口
4. `Orchestrator`
5. `Context Builder`
6. `Trace`
7. `JSON Analyzer`
8. `Hierarchical Summarizer`
9. `Module Registry / Memory Policy / Planner / 聚感 / 权限层`
10. 之后才是 NAS、家居、视觉、多模型路由等增强能力

---

## 11. 何时允许扩编到 5号、6号

当前默认先以 4 个 Codex 编制推进。

只有在满足以下条件后，才考虑扩编：

1. 1号 / 3号 / 4号 开始出现明显空转
2. 2号 回填稳定跟得上
3. 同一核心文件不再频繁撞 owner
4. 当前主线边界已较清晰
5. 底座统一程度进一步提高

扩编原则：
- 先确保边界稳定，再加人
- 不是为了“更多线程”，而是为了“更高吞吐量且不返工”

---

## 12. 每晚收尾动作

每晚推进结束前，默认完成以下 3 件事：

1. 确认今天收了哪几张工单
2. 确认当前唯一施工工单是什么
3. 确认下一张正式工单是什么

若有代码实施完成，默认同步：
- `NEXT_ACTION.md`
- `TECH_DEBT.md`
- 当前主链结论摘要

---

## 13. 当前流程定位

当前流程不属于临时式推进，而属于：

- 主线程驱动
- 工单驱动
- 最小验证驱动
- 台账同步驱动

当前流程目标不是“多做功能”，而是：

- 保证方向正确
- 保证边界清楚
- 保证主链持续收口
- 保证后续扩展建立在统一底座上

---

## 14. 一句话定盘

三花聚顶当前采用的是：

**总控中枢 + 主施工位 + 记录位 + 审计位 + 验证位**

的协同推进方式。

任何新工单、新线程、新能力接入，默认都必须服从：

- 主线程优先
- 文件 owner 清晰
- 控制面同步
- 边界不漂移
- 验收先于扩张
