# TECH_DEBT

## 使用规则
1. 发现问题先登记，再决定是否当前修
2. 技术债描述要短、准、可追踪
3. 修复后必须回填状态和验证结果

---

## 当前技术债

### TD-001
- 标题：GUI 仍承担过多系统编排职责
- 等级：高
- 范围：`entry/gui_entry/gui_main.py`
- 现象：
  - 存在预路由、记忆短路、AICore 桥接、alias 强制加载等职责堆叠
- 影响：
  - GUI 不再只是 Shell，系统真相源边界模糊
- 建议：
  - 继续剥离到统一总控/桥接层

---

### TD-002
- 标题：动作调用链仍需统一口径
- 等级：高
- 范围：AICore / Dispatcher / GUI Bridge
- 现象：
  - 不同入口可能存在多种调用路径
- 影响：
  - 调试难、追踪难、验收难
- 建议：
  - 统一以 `dispatch_action / call_action` 为主调用口

---

### TD-003
- 标题：部分历史资产和副本文件仍混在主仓
- 等级：中
- 范围：docs、旧模块副本、txt 辅助文件
- 现象：
  - 存在 `_副本`、`.txt`、旧结构残留
- 影响：
  - 噪音大，影响施工判断
- 建议：
  - 后续做一次“主仓资产瘦身”和归档策略整理

---

### TD-004
- 标题：验证证据与工单结果回填还未形成强约束
- 等级：中
- 范围：全局协作流程
- 现象：
  - 容易出现“改了但没证据”
- 影响：
  - 验收质量波动
- 建议：
  - 让 4号验证位标准化输出日志、命令、结果摘要
